## /stock-mcap-history 接口报 "JSON Parse error: Unrecognized token '<'" 修复

**症状**：浏览器控制台 `[mcap] failed – "601888" – JSON Parse error: Unrecognized token '<'`，Vite 控制台有 `[ECharts] Specified 'grid.containLabel'...` 警告。市值线始终不画。

**根因**：新增后端端点 `/stock-mcap-history` 时，**漏把它加到 `frontend/vite.config.ts` 的 `PROXY_PATHS` 列表**。Vite SPA 模式下，命中不到 proxy 的路径会被 fallback 到 `index.html`（看 vite 的 `appType: "spa"`），所以 fetch 收到 HTML 文档，JSON.parse 抛错。

**修复**：`vite.config.ts` 第 20 行后加一行 `"/stock-mcap-history",`。但 vite.config 改动**不会触发 Vite 自动重启**（只 HMR 业务代码），必须 kill 进程让 launchctl KeepAlive respawn。

**操作**：
1. `lsof -iTCP:5173 -sTCP:LISTEN -n -P` 找到 vite PID
2. `kill <PID>` → launchctl KeepAlive 拉起新进程
3. `curl -i http://127.0.0.1:5173/stock-mcap-history?code=601888&market=A` 验证 200 + application/json
4. 提示用户 ⌘+Shift+R 硬刷新

**教训**：以后加任何后端新端点，凡是前端要调用的，**必须**先在 vite.config 的 `PROXY_PATHS` 数组里登记，否则 Vite 默认 fallback 到 SPA index.html。可以在 PR checklist 里加一条"新端点 = proxy + 后端 route + api.ts client"三件套。

## 营收柱的"稀疏显示"修正

**用户反馈**：截图显示 2025-07-11 这种非季报日的周也出现 533.60 亿 TTM 柱子（=2025Q2 TTM 沿用到了 7 月、8 月…），但用户语义是：**季报日（03-31/06-30/09-30/12-31）所在周**才画一根柱子，其他周是空。

**之前错误的理解**：把营收 TTM 沿 X 轴=周做 forward-fill（最近一季连续保留到下一季报日）。这导致半年里每天都显示同一个 TTM 值，"季"的概念被稀释。

**修正后逻辑**（`lib/stock-analytics.ts:285`）：
```ts
const revenue: (number | null)[] = new Array(weekly.length).fill(null);
for (const pr of periodList) {  // pr.period = 季报日 06-30 / 12-31
  // 找「<= 季报日」的最末一个周聚合索引
  let lastIdx = -1;
  for (let i = 0; i < weekly.length; i++) {
    if (weekly[i].date <= pr.period) lastIdx = i;
    else break;
  }
  if (lastIdx >= 0) revenue[lastIdx] = pr.rev;
}
```

效果：8.5 年共 34 季 → 34 根柱子，分布在 34 个不同的"季报日所在周"上（如 2025-06-27 周、2025-10-03 周、2025-12-26 周、2026-04-03 周），其他 ~400 周都是 null（柱空）。市值线照常画 434 根。

**柱宽**：`barWidth: 6`（之前按 dates.length 动态 2-14，现在固定 6）。
